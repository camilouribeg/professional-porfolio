The Problem

1,500 clients. 3 files each. All different.

Each audit client provides a General Ledger, Trial Balance, and PDF journal documents — sourced from different ERP systems, with inconsistent schemas, missing values, and reconciliation errors that must be detected and resolved before the data can be used.

📊

General Ledger (GL)

Every individual transaction for the financial year. Each entry has a debit and credit side — they must always balance. This is the source of truth.

CSV / Excel
📋

Trial Balance (TB)

A summary: total debits and credits per account for the period. GL rolled up by account must equal TB exactly — if not, there is a variance that requires investigation.

CSV / Excel
📄

Journal PDFs

Supporting documentation — scanned or digital. Unstructured. Requires OCR and LLM extraction to recover structured transaction data.

PDF

5 problems in the data

✗Column name variance — "AccountCode" in SAP, "Acc_Ref" in MYOB, "acct_no" in Xero. Same field, hundreds of names across 1,500 clients.
✗Missing critical values — nulls in account codes, amounts, or dates. A transaction with no account code cannot be posted or reconciled.
✗Duplicate transactions — same entry posted twice. Exact duplicates caught by hash; near-duplicates (same amount ±3 days) flagged by AI for human review.
✗GL ≠ Trial Balance — rolled-up GL totals per account don't match the TB balance. Every non-zero variance is a reconciliation failure requiring explanation.
✗Format variance — different delimiters, encodings (UTF-8 vs Latin-1), date formats (2024-01-15 vs 15/01/24), and multi-sheet Excel structures.

System Architecture

7-layer pipeline — how the data flows

Each layer has a single, clear responsibility. AI nodes are used exactly where ambiguity exists and nowhere else. Every decision is logged with a rule ID for auditability.

01 — Ingest & Land

Chain of Custody Starts Here

Files arrive via SFTP or Azure Blob. Before anything else, each file receives a SHA-256 hash — this proves the file that was analysed is exactly the file the client sent. Files are written write-once to Azure WORM storage and never modified.

# Chain of custody
hash = sha256(file_bytes) # tamper-evident fingerprint
raw_zone.write_once(file, metadata={"hash": hash, "client_id": id})
SHA-256 per file Azure Blob / SFTP WORM write-once Postgres control table 100% deterministic

02 — Classify & Profile

ERP Detection + File Type Routing

Is this a GL, TB, or PDF journal? Which ERP produced it — SAP, Oracle, Xero, MYOB, NetSuite? Deterministic first: filename patterns and known header signatures cover ~80% of files. AI fallback for unknown shapes, with a confidence score on every output.

Known ERP → skip LLM

SAP / Oracle / Xero / MYOB / NetSuite headers matched deterministically

Unknown → LLM classify

confidence ≥ 0.95 → auto-route · below threshold → human review queue

Det. first AI fallback confidence_score on every output Saves LLM cost on 80% of files

03a — CSV / Excel Extract

Deterministic Parsing

Tabular data with defined structure. No AI needed — just correct parsing. Handle encoding variance (UTF-8, Latin-1, CP1252), dtype coercion (amounts to float, dates to datetime), and multi-sheet Excel workbooks.

df = pd.read_csv(file,

Deterministic Encoding detection Type coercion

03b — PDF Journal Extract

OCR + LLM + Pydantic

Unstructured. This is where AI earns its place. Digital PDFs go through pdfplumber; scanned documents through Azure Document Intelligence OCR. The LLM then extracts structured fields, validated immediately by Pydantic schema enforcement.

pdfplumber (digital) Azure Document Intelligence (OCR) Pydantic validated Retry on fail → human queue

04 — Schema Mapping → Common Data Model

AI proposes once · Human approves once · Cache decides forever

Every ERP exports the same fields with different column names. The pipeline maps everything to a single CDM. Known ERP templates are cached and deterministic — zero LLM cost. Novel columns go to the LLM for a mapping proposal, then to a human analyst for one-time approval. The approved mapping becomes versioned config: deterministic forever.

1. Known ERP template

SAP / Xero / MYOB → cached config, zero LLM

2. LLM proposes mapping

Novel column + confidence → human queue

3. Human approves once

Saved as versioned config → deterministic from engagement 2+

Reusable AI skill LLM cost paid once per schema variant Version-pinned per run Config inheritance: Global → ERP template → Client override

05 — Quality Validation & GL ↔ TB Reconciliation

The most critical layer — zero AI in the decision path

Every rule is deterministic and logged with a rule ID. The GL vs TB reconciliation is pure arithmetic — AI never computes a financial figure. Variances go to human review; AI can generate a candidate explanation, annotated as advisory only.

Pandera Validation Rules

RULE-001 — Required fields present RULE-002 — Date within financial year RULE-003 — DR = CR per journal entry RULE-004 — Exact duplicate detection (hash) RULE-005 — Fuzzy near-duplicate flag (AI)

GL ↔ TB Reconciliation

# Roll up GL per account
gl_agg = gl.groupby('account')
           [['debit','credit']].sum()

variance = gl_agg - tb_balance
flags = variance[variance != 0]
100% deterministic decisions Every rule logged with rule ID Variance → human review queue AI explanation = advisory annotation only
06 — LangGraph Orchestration

Stateful Graph — Every Client Through the Same Auditable Pipeline

LangGraph wraps the entire pipeline as a stateful directed graph. Each node above is a graph node. Conditional edges route based on confidence scores and validation results. The graph can pause, persist state, and resume — enabling human-in-the-loop review without restarting from scratch.

ingest Hash + land files in WORM storage Deterministic
classify ERP + file type detection with confidence score AI node
route_by_type CSV path vs PDF path — conditional branch Deterministic
extract_pdf OCR + LLM extraction + Pydantic validation AI node
map_schema Column → CDM mapping (cached or LLM proposal) AI → Cache
validate Pandera rules — all deterministic, logged by rule ID Deterministic
reconcile GL vs TB arithmetic — variance calculation Deterministic
human_review interrupt() — pause, persist state, await analyst input Human-in-loop
persist Append-only audit log + clean CDM output Deterministic

Conditional Edges

# After classify:
confidence ≥ 0.95  → route_by_type
confidence < 0.95  → human_review
# After extract_pdf:
pydantic OK        → map_schema
fail + retries < 3 → LOOP back (retry with error fed as context)
fail + retries ≥ 3 → human_review
# After reconcile:
variance ≤ tolerance → persist
variance > tolerance → human_review
▸Stateful — knows where each of the 1,500 clients is in the pipeline; crash-safe checkpointing
▸Loops — retry failed PDF extractions with the previous error as context
▸Human interrupt — pause, persist full state, resume on analyst sign-off
▸Parallel fan-out — stream 1,500 clients concurrently, isolated per thread_id

Design Decisions

Every component classified — AI or deterministic?

Nothing AI-driven without a clear justification. The rule: if a regulator could challenge it, deterministic code with a logged rule ID handles it.
