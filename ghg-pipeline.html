System Architecture

How the pipeline works

Data sources and hierarchy configuration are independent starting points — both feed into the automation layer, which maps everything before calculation. Results then fork into aggregation, forecasting, and reporting.

5 Dimensions 🏭 Sector · ⚙️ Scope 1 / 2 / 3 · 💨 Emission Source · 🧪 Gas Type · 🕐 Time Window — every result is indexed across all five

03 — Client Automation Layer

Bespoke ETL per Client

Each client gets a dedicated Python pipeline that knows its data structure, naming conventions, and quirks. Pulls from all connected sources, joins records using shared identifiers (e.g. API well numbers), filters by the configured time window, normalises units, and outputs clean typed CSVs ready for the calculation engine.

Pandas transforms Entity resolution Time window filter Unit normalisation Throughput calculation Per-client config

04 — Input Mapping & Synchronisation

Map Everything to the Grid

Every data point is mapped to a specific position in the 5D grid before calculation begins. This is where the hierarchy, time window, and industry segment from step 02 are applied — locking every record to a facility level, a scope assignment, and a reporting period.

05 — Input Aggregation

Combine Facilities Before Calculation

When a client wants a single combined inventory across multiple facilities, inputs are merged here before the engine runs. The hierarchy defined in step 02 controls which facilities are combined and at what level. If not needed, this step is bypassed and each facility runs independently.

Merge facility inputs Weight-average EFs Sum activity data Respects hierarchy config
06 — Calculation Engine

Plugin Architecture — One Module Per Sector

The dispatcher reads the industry segment and routes to the correct module. Every module has an identical interface — add a new sector by writing one new module. The engine, EF registry, and GWP conversion never change.

Extensibility

One framework, any industry

Extending to mining, manufacturing, agriculture, or any other sector follows the same five steps — every time. The core engine never changes.

1

Regulatory Mapping

Identify applicable standards, IPCC source categories, and available emission factors before writing any code.

2

Data Ingestion

Map sector-specific data sources — mine records, blast logs, SCADA — and build ingestion connectors.

3

EF Registry Extension

Add sector emission factors with version tags so annual regulatory updates flow through without touching calculation code.

4

Calculation Module

Write one new plugin module with the same interface as all existing modules. Sector logic stays fully isolated.

5

Validation & Output

Sector-specific QA rules. Every output traceable to input version, EF version, and calculation method used.
